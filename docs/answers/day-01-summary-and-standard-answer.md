# Day 01 - 学习总结与标准答案

## 今日是否完成

- 程序真正的入口是 `_start`，不是 `main`
- 没有 libc 也可以生成和运行 ELF 程序
- `printf` 不是起点，而是建立在更底层运行时能力之上的结果

## 今日总结

今天的重点不是实现功能，而是校正程序启动模型。

在普通 C 开发里，`main` 很容易被误认为是程序入口，但在 Linux 可执行文件里，内核真正跳转到的是 ELF 入口点，通常对应 `_start`。在宿主环境下，这个入口通常由 CRT 启动代码提供；在本项目中，我们会自己实现它。

同时也要建立第二个关键认识：没有 libc 并不意味着“不能写 C 程序”。只要有编译器、汇编器、链接器，并且我们自己提供程序入口和最小运行时，仍然可以生成 ELF 可执行文件。只有当程序需要与内核交互时，才需要自己封装 syscall。

最后，`printf` 不能作为第一步，不是因为它“只是调用了几个 syscall”，而是因为它依赖更底层的一整套能力，包括输出后端、字符串处理、可变参数、格式解析和整数格式化。因此我们必须从 `_start` 和最小 syscall 开始，自底向上搭建。

## 标准答案

### 1. 没有 libc 时，程序从哪里开始？

标准答案：

程序不是从 `main` 开始，而是从 ELF 入口点开始，通常这个入口符号叫 `_start`。在常规环境中，这个入口通常由 CRT 启动代码提供；在本项目中，我们会自己实现 `_start`，由它接住内核交来的控制权。

### 2. 为什么不能直接从 `printf` 开始？

标准答案：

因为 `printf` 不是底层起点，而是建立在更底层运行时能力之上的高层接口。它至少依赖输出能力、字符串处理、可变参数、格式解析和整数转字符串等基础组件，所以必须等这些能力逐步具备后才能实现。

### 3. `-nostdlib` 的作用是什么？

标准答案：

`-nostdlib` 用来禁止链接阶段自动链接宿主标准库和标准启动文件，例如 glibc 和 CRT 相关组件。这样程序就不会偷偷依赖宿主运行时，我们必须自己提供入口和最小运行时支持。

### 4. `-ffreestanding` 的作用是什么？

标准答案：

`-ffreestanding` 用来告诉编译器：当前程序运行在一个非普通 hosted 环境中，不能默认依赖完整标准运行时。它的核心意义是让编译器按“自举环境、内核环境、裸机环境、运行时环境”的视角处理当前代码，而不是假设这是一个已经具备完整 libc 支持的普通用户程序。

### 5. 没有 libc，还能不能生成 ELF 可执行文件？

标准答案：

可以。只要有编译器、汇编器和链接器，并且我们自己提供程序入口和最小运行时，就仍然可以生成并运行 ELF 程序。只有在需要与内核交互时，才需要自己实现 syscall 封装。

### 6. `_start` 和 `main` 的职责区别是什么？

标准答案：

- `_start` 是程序的真实入口，负责接住内核控制权并建立最小执行路径
- `main` 不是入口，而是运行时调用的程序主体逻辑入口

一句话概括：

`_start` 负责把程序“带起来”，`main` 负责执行程序主体逻辑。

### 7. `_start` 最少需要做哪两件事，程序才能正常结束？

标准答案：

在本项目的最小模型里，`_start` 至少要做两件事：

- 调用你的 C 入口函数
- 在 C 入口返回后，通过 `exit` syscall 把返回值交还给内核，显式结束进程

这里的关键点不是“释放很多资源”，而是确保程序有一条合法、明确的退出路径，而不是执行流跑飞。

## 今日最小知识闭环

如果你今天只记住 6 句话，应该记住这些：

- 程序真正的入口是 `_start`，不是 `main`
- `_start` 通常由 CRT 提供，但在这个项目里要自己实现
- 没有 libc 也可以生成 ELF 程序
- 与内核交互时需要自己封装 syscall
- `printf` 是阶段性结果，不是起点
- `-nostdlib` 和 `-ffreestanding` 的目标都是切断对宿主完整运行时的默认依赖
