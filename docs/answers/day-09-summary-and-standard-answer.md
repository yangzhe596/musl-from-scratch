## 标准答案

### 1. 测试覆盖了哪些关键点？

标准答案：

测试覆盖了以下关键点：
- **空串**：`test_strlen("")` 测试空字符串长度为 0
- **边界长度**：`test_strlen("1234567890")` 测试较长的字符串
- **对齐与非对齐地址**：`test_memset` 中测试了对齐地址（`buf`）和非对齐地址（`buf + 1`）
- **`n == 0`**：`test_memset` 和 `test_memcpy` 中都测试了 `n = 0` 的情况

### 2. 语义边界是什么？

标准答案：

- `strlen`：返回不包含终止符 `'\0'` 的长度，依赖输入必须是合法 C 字符串
- `memset`：按 `unsigned char` 填充，返回原始 `dst` 指针
- `memcpy`：不处理重叠内存，返回原始 `dst` 指针
- `memmove`：可以正确处理重叠内存，返回原始 `dst` 指针

### 3. 优化边界是什么？

标准答案：

- `memset`：可以使用三步走优化（先对齐，再按 `size_t` 块填充，最后处理尾巴）
- `memcpy`：优化更复杂，需要考虑源和目的的对齐状态，只有对齐到相同边界才能按 `size_t` 块复制
- `memmove`：需要根据重叠方向选择复制顺序，优化更复杂

### 4. 哪些场景不能乱用？

标准答案：

- `memcpy` 不能处理重叠内存，如果源和目的重叠，需要使用 `memmove`
- `memset` 的 `int c` 参数会转换为 `unsigned char`，不能直接传递负数期望得到特定结果
- `strlen` 依赖输入必须是合法 C 字符串，对 `NULL` 调用 `strlen` 是未定义行为

### 5. `-fno-builtin` 为什么重要？

标准答案：

`-fno-builtin` 防止编译器将我们的函数调用优化为内联版本。如果没有这个选项，编译器可能会将 `strlen`、`memset`、`memcpy` 等函数调用替换为内联版本，这样我们测试的就不是我们自己的实现，而是编译器的内联版本。

### 6. 如果后面实现 `printf`，会不会复用今天写的这些基础函数？

标准答案：

会。`printf` 的实现需要：
- `strlen`：计算字符串长度
- `memset`：初始化缓冲区
- `memcpy`：复制字符串
- `memmove`：处理可能重叠的内存操作

这些基础函数是 `printf` 实现的基石。

### 7. 为什么基础函数越底层，越要先把语义边界钉死？

标准答案：

因为底层函数会被高层函数依赖。如果底层函数的语义不清晰或有歧义，高层函数的实现就会出问题。例如：
- 如果 `memcpy` 的重叠处理不明确，`memmove` 的实现就会混乱
- 如果 `memset` 的 `int c` 参数处理不明确，填充结果可能不符合预期
- 如果 `strlen` 的返回值定义不清晰，字符串处理函数就会出错
