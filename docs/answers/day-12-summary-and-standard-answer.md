## 今日总结

今天的重点是实现最小 `mini_printf`，串起格式解析、参数读取和输出。

在实现方面，`mini_printf` 的核心是一个简单的状态机：
1. 遍历格式串，普通字符直接输出
2. 遇到 `%` 进入格式处理
3. 根据格式字符（`%s`/`%d`/`%x`/`%%`）分发到对应的输出函数
4. 未知格式符原样输出 `%` + 字符

关键设计：
1. `va_arg` 类型必须匹配：`%d` 取 `int`，`%x` 取 `unsigned int`，取错类型会默默给错误值
2. `%s` 对空指针的处理：输出 `(null)` 是 musl/glibc 的惯例，方便调试
3. `default` 分支：未知格式符原样输出，不崩溃，这是防御性设计
4. `written` 返回值：现在 `%d` 和 `%x` 没更新 `written`，等做 `sprintf` 时再补

踩的坑：
- `%d` 不支持 `long`：`va_arg(ap, int)` 只读 4 字节，传 `long` 会丢高位
- 解决：后续加 `%ld` 支持

## 标准答案

### 1. mini_printf 的主循环逻辑是什么？

标准答案：

```
遍历 fmt 字符串
├── 普通字符 → 直接输出，written++
├── 遇到 '%' → 进入格式处理
│   ├── '%' → 输出一个 '%'
│   ├── 's' → va_arg 取 const char*，逐字符输出
│   ├── 'd' → va_arg 取 int，调用 put_int
│   ├── 'x' → va_arg 取 unsigned int，调用 put_hex
│   └── 其他 → 原样输出 '%' + 未知字符
└── fmt++ 继续扫描
```

### 2. 为什么 %d 不支持 long？

标准答案：

`%d` 对应 `va_arg(ap, int)`，在 x86_64 上 `int` 是 4 字节。如果传入 `long`（8 字节），`va_arg` 只取前 4 字节，高位直接丢失，不会报错。

解决方法是支持 `%ld`，对应 `va_arg(ap, long)`。格式串里的长度修饰符（`l`、`ll`、`h`）就是用来告诉 `va_arg` 该取多大的。

### 3. 未知格式符应该怎么处理？

标准答案：

标准做法是原样输出 `%` + 未知字符。比如 `%q` 会输出 `%q`。

这样做的好处：
- 不崩溃，程序继续运行
- 用户能看到是哪个格式符不支持
- 和 musl/glibc 的行为一致

工业级 `printf` 还会设置 `errno = EINVAL`，但现在不需要。
