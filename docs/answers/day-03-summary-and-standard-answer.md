# Day 03 - 学习总结与标准答案

## 今日是否完成

Day 3 可以视为完成。

今天的目标不是让程序“正常退出”，而是亲手把最小启动路径搭起来：自己提供 `_start`，从汇编进入 C 函数，并确认返回值会按照 ABI 规则留在返回寄存器里。

## 今日总结

今天最关键的进展是：你第一次自己接住了程序真实入口。

在没有宿主 CRT 的情况下，内核不会帮你调用 `main`，也不会自动处理返回路径。你必须自己定义 `_start`，并在里面显式调用你的 C 入口函数，例如 `c_main`。

这里 `_start` 调用 `c_main` 时，遵守的不是 syscall 约定，而是普通 x86_64 System V 函数调用约定。也就是说，`call c_main` 之后，C 函数的返回值会放在 `rax`，对 `int` 返回值来说，通常直接看 `eax` 即可。

今天先不接 `exit` 是合理的。教学目标是先把 `_start -> c_main` 这条链路打通，再在下一天把返回值交给 `sys_exit`。这样能把“函数调用 ABI”和“syscall ABI”分开理解，不会混在一起。

## 标准答案

### 1. Day 3 最小实现到底要完成什么？

标准答案：

Day 3 的最小实现只需要完成三件事：

- 自己定义程序真实入口 `_start`
- 在 `_start` 中调用 C 入口函数 `c_main`
- 确认 `c_main` 的返回值已经按 ABI 放在返回寄存器里

不要求今天就把退出路径补完整。

### 2. `_start` 调用 `c_main` 时遵守哪套调用约定？

标准答案：

遵守 x86_64 System V ABI，也就是普通用户态函数调用约定，而不是 Linux syscall 约定。

这意味着：

- 函数参数按普通函数调用规则传递
- `call` 会压入返回地址
- 返回值放在 `rax`

### 3. `c_main` 返回后，返回值在哪个寄存器里？

标准答案：

返回值在 `rax`。如果返回类型是 `int`，通常可以直接理解为结果在 `eax`。

### 4. 为什么今天不急着处理 `argc`、`argv` 和 `envp`？

标准答案：

因为 Day 3 的目标只是先打通最短控制流路径，确认：

- `_start` 是真实入口
- `_start` 可以主动调用你的 C 函数
- C 函数返回值会回到 `_start`

如果一开始就把参数解析、栈布局和退出路径全部混进来，学习重点会被冲散。

### 5. 为什么 `_start` 不能像普通函数那样随便 `ret`？

标准答案：

因为 `_start` 不是被某个普通调用者通过 `call` 调起来的。它是进程入口，没有一个正常的上层返回地址可供 `ret` 使用。直接 `ret` 会让执行流跳到无意义的位置，通常导致崩溃或未定义行为。

### 6. Day 3 和 Day 4 的边界是什么？

标准答案：

- Day 3 解决“如何从 `_start` 进入你自己的 C 代码”
- Day 4 解决“如何把 C 返回值交回给内核并完成退出”

也可以理解为：

- Day 3 学函数调用 ABI
- Day 4 学 syscall ABI

## 今日最小知识闭环

如果今天只记住 5 句话，应该记住这些：

- 程序真实入口是 `_start`
- 没有 CRT 时，`_start` 要自己写
- `_start` 调 `c_main` 时遵守普通函数调用 ABI
- `c_main` 的返回值会放在 `rax/eax`
- 退出路径留到下一天用 `exit` syscall 接上
