## 今日总结

今天的重点是阶段四的收尾和局限性分析。

在测试方面，我们已经覆盖：
- `malloc(0)` → 返回 NULL
- `free(NULL)` → 不崩溃
- 小块复用 → free 后 malloc 同大小能复用
- 大块分裂 → 大块能分裂成小块
- 相邻合并 → 相邻空闲块能合并

在局限性方面，当前分配器的问题：
1. **碎片**：只能合并相邻块，无法整理远距离碎片
2. **查找复杂度**：first-fit 是 O(n)，链表长了性能差
3. **元数据安全性**：没有 magic number，无法检测 double free
4. **小对象开销**：每个块 32 字节 header，小对象场景占比高
5. **线程安全**：没有锁，多线程会出问题

与 musl 的对比：
- musl 的 oldmalloc 也是 free list，但有更完善的合并和碎片整理
- musl 的 mallocng 用分槽（size class）解决小对象问题，O(1) 分配

## 标准答案

### 1. 为什么简单 allocator 在小对象场景下表现不好？

标准答案：

小对象（如 16 字节）的问题：
- 每个块有 32 字节 header 开销，占比 200%
- free_list 长度随分配次数线性增长
- 每次 malloc 要遍历链表找空闲块，O(n)
- 10000 次 malloc(16) → 链表 10000 节点，每次遍历平均 5000 次

对比 mallocng 的分槽方案：
- 按大小分组（16、32、48...）
- 每组有自己的 free list
- 分配时直接取对应组的第一个空闲块，O(1)

### 2. 如果要提升性能，先优化哪一层？

标准答案：

三层开销对比：
- 用户接口（malloc/free）：只是函数调用，几乎无开销
- **块管理（free_list/search/split/merge）：遍历链表 O(n)，是主要瓶颈**
- 系统调用（mmap）：只在没有空闲块时才调用，有 free list 后很少触发

所以优化重点在块管理层：
- 用分槽（size class）替代线性链表
- 用位图或 buddy system 加速查找
- 用 thread-local cache 减少锁竞争

这就是 Stage 5 mallocng 要做的事。
