## 今日总结

今天的重点是合并逻辑和分配器的调试性。

在实现方面，`merge_free` 的逻辑：
- 遍历 free list，检查当前块和下一块是否物理相邻
- 如果相邻且都 free，合并成一个大块
- 合并后继续检查，可能连续合并多个块

关键收获：
1. 合并的前提是"物理相邻"，不是"链表相邻"
2. use-after-free 不会立刻崩，因为内存还在进程地址空间里
3. double free 本身不崩（free=1 是幂等操作），危险的是"释放后复用再释放"
4. 调试器不是可选项——没有 heap_dump 几乎无法调试分配器 bug

理解的 bug 模式：
- use-after-free：写入已释放的块，可能覆盖新用户的数据
- double free：块被复用后，旧指针再次 free 破坏新用户的元数据
- 这些 bug 不会立刻崩，可能在完全不相关的代码里才暴露

## 标准答案

### 1. 为什么 allocator bug 不会立刻崩？

标准答案：

free 只是标记 free=1，不改变页表映射：
- 内存还在进程地址空间里
- 内核不知道你 free 了
- 写入不会触发 segfault

bug 只有在以下情况才暴露：
- 块被另一个 malloc 复用
- 两个指针同时认为自己拥有这块内存
- 数据被覆盖，逻辑错乱
- 可能在完全不相关的代码里才崩

这就是为什么 allocator bug 是最难调试的 bug 之一。

### 2. double free 会怎样？

标准答案：

当前实现 double free 不会立刻崩：
- `hdr->free = 1` 是幂等操作，写两次和写一次一样
- `merge_free` 只合并相邻且都 free 的块，不会重复合并

但危险场景是：
```
1. free(A) → A 标记 free
2. malloc(复用 A) → 用户写数据
3. free(A) → A.free = 1，破坏新用户的元数据
```

工业级分配器会在块头加 magic number 或 generation counter 检测这种情况。
